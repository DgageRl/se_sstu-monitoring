## 1. Суть кейса (что строим)

### 1.1. Цель системы (миссия)

Обеспечить сбор, хранение и анализ метрик, логов и трейсов с сервисов продуктовой платформы, предоставляя единую панель наблюдения, интеллектуальную систему алертинга с эскалацией и отчётность по SLO для минимизации времени обнаружения и устранения инцидентов (MTTR, MTTA).

### 1.2. Контекст (среда и внешние системы)

#### 2.1. Пользователи — описание поведения

| Роль | Поведение |
| :--- | :--- |
| **SRE-инженер** | Настраивает правила алертинга, создаёт дашборды, анализирует метрики/логи/трейсы, управляет силенсами. |
| **Дежурный инженер (On-Call)** | Получает уведомления об инцидентах, подтверждает их (Ack), исследует причины по дашбордам, фиксирует митигацию. |
| **Тимлид платформы** | Просматривает отчёты по инцидентам и SLO, настраивает политики эскалации, получает эскалации при долгом отсутствии реакции. |
| **Владелец сервиса** | Получает ежемесячные отчёты о доступности своего сервиса, участвует в постмортемах. |

#### 2.2. Исполнители — описание поведения

| Исполнитель | Поведение |
| :--- | :--- |
| **Сервисы платформы** | Экспортируют метрики в формате Prometheus, отправляют логи (JSON, structured) через агент (Vector/Fluentd), отправляют трейсы по OTLP. |
| **Агент сбора (на узлах)** | Собирает метрики node (CPU, RAM, disk, network), пересылает их в систему. |

#### 2.3. Интеграции (на выбор команды)

| Интеграция | Назначение |
| :--- | :--- |
| **Slack** | Доставка уведомлений об инцидентах, кнопки Ack/Silence, дайджесты. |
| **PagerDuty** | Управление on-call расписанием, эскалация, синхронизация инцидентов. |
| **Jira** | Автоматическое создание тикетов при эскалации инцидентов. |
| **IdP (Keycloak)** | Аутентификация и RBAC для доступа к UI и API. |

### 1.3. Границы SoI

#### 3.1. Внутри системы

- Сервис приёма метрик (Metrics Ingester)
- Сервис приёма логов (Logs Ingester)
- Сервис приёма трейсов (Traces Ingester)
- Движок алертинга (Alert Evaluator)
- Менеджер уведомлений (Alertmanager + Notifier)
- Модуль инцидент-менеджмента (легковесный)
- База данных конфигураций (правила, дашборды, политики)
- Модуль отчётности по SLO
- Административный веб-интерфейс (Grafana + самописный UI)

#### 3.2. Снаружи системы

- IdP/SSO (Keycloak) — аутентификация вынесена
- Почтовый сервер (опционально, fallback)
- Внешние системы: Slack, PagerDuty, Jira
- BI/аналитика (опционально, экспорт метрик)

## 2. Роли и сценарии (CONOPS)

### 2.1. Роли (по модели CONOPS)

| Роль | Описание |
| :--- | :--- |
| **SRE** | Настройка и поддержка системы мониторинга. |
| **Дежурный инженер** | Реагирование на инциденты. |
| **Тимлид** | Управление эскалацией, анализ отчётности. |
| **Владелец сервиса** | Потребление SLO-отчётов. |

### 2.2. Сценарии (5 шт.)

<details>
<summary><b>📋 Сценарий 1. Создание и валидация правила алерта (SRE)</b></summary>

**Цель:** Настроить алерт на превышение ошибок в сервисе платежей.

**Условия начала:** SRE авторизован в UI, имеет роль Editor.

**Основной поток:**
1. SRE открывает раздел «Правила алертинга».
2. Нажимает «Создать правило».
3. Вводит PromQL-запрос: `rate(http_errors_total[5m]) > 0.01`.
4. Указывает severity: `critical`, интервал: `30s`, duration: `5m`.
5. Добавляет аннотации: `summary`, `dashboard_url`, `runbook_url`.
6. Нажимает «Сохранить».
7. Система валидирует синтаксис запроса.
8. Система деплоит правило в движок алертинга.
9. UI отображает статус «Активно».

**Результат:** Правило сохранено и начнёт проверяться в следующем цикле.

**Исключения:**
- *Невалидный запрос:* Система возвращает ошибку с подсветкой места.
- *Нет прав:* Блокировка, уведомление «Требуется роль Editor».

**Режимы деградации:** БД конфигураций недоступна → использование локального кеша последней синхронизации с Git-репозитория (GitOps).

</details>

<details>
<summary><b>🔔 Сценарий 2. Срабатывание алерта и уведомление дежурного</b></summary>

**Цель:** Инженер получает уведомление о проблеме.

**Условия начала:** Активно правило, метрики превысили порог на 5 минут.

**Основной поток:**
1. Движок алертинга обнаруживает нарушение: `firing`.
2. Alertmanager получает алерт.
3. Применяются группировка и подавление (inhibition).
4. Нотификатор определяет текущего on-call инженера через PagerDuty API.
5. Нотификатор отправляет сообщение в Slack: заголовок, описание, ссылки, кнопки.
6. Дежурный видит уведомление в канале `#incidents`.

**Результат:** Инцидент виден, инженер уведомлён.

**Исключения:**
- *Slack недоступен:* Отправка в Telegram + email как fallback.
- *PagerDuty API недоступен:* Использовать кешированное расписание.

</details>

<details>
<summary><b>🩺 Сценарий 3. Подтверждение инцидента и диагностика</b></summary>

**Цель:** Инженер начинает работу, получает контекст.

**Основной поток:**
1. Инженер нажимает «Acknowledge» в Slack.
2. Система получает webhook, обновляет статус инцидента.
3. Инженер переходит по ссылке `dashboard_url`.
4. Система (Grafana) открывает дашборд с предзаполненными labels инцидента.
5. Инженер анализирует графики ошибок, переходит в раздел логов, затем в трейсы.
6. Инженер идентифицирует корневую причину.

**Результат:** Причина найдена, начато устранение.

</details>

<details>
<summary><b>📈 Сценарий 4. Эскалация инцидента (не подтверждён)</b></summary>

**Цель:** Привлечь тимлида, если инженер не отреагировал.

**Условия начала:** Алерт в статусе `firing`, не подтверждён 10 минут.

**Основной поток:**
1. Система запускает таймер эскалации.
2. Нотификатор отправляет уведомление тимлиду в Slack.
3. Система создаёт тикет в Jira с полями: summary, description, priority=High.
4. Тимлид назначает ответственного в Jira.
5. При резолве алерта система автоматически закрывает тикет.

**Результат:** Инцидент не потерян, создан тикет.

</details>

<details>
<summary><b>📊 Сценарий 5. Формирование отчёта об инцидентах и SLO</b></summary>

**Цель:** Получить статистику за месяц.

**Основной поток:**
1. Владелец выбирает в UI сервис `payment-api` и период «Март 2026».
2. Система запрашивает данные инцидентов из внутреннего хранилища и Jira.
3. Система запрашивает SLO-метрики (доступность, latency) из базы метрик.
4. Генерируется отчёт: MTTA, MTTR, кол-во инцидентов, аптайм, соответствие SLO (99.9%).
5. Владелец скачивает PDF.

**Результат:** Отчёт сформирован.

</details>

### 2.3. Сводная таблица сценариев, сбоев, деградации

| № | Сценарий | Критичный сбой | Деградация (fallback) |
|:--:|:---|:---|:---|
| 1 | Создание правила | БД конфигураций недоступна | GitOps: кеш последней синхронизации |
| 2 | Срабатывание алерта | Slack недоступен | Telegram + email fallback |
| 3 | Диагностика | Grafana недоступна | CLI запросы к Query API |
| 4 | Эскалация | Jira недоступна | Локальная очередь + email отчёт |
| 5 | Отчётность | SLO-метрики недоступны | Отчёт только по инцидентам |

| ID | Требование | Метрика | Метод проверки |
|:---|:---|:---|:---|

| NFR-2 | Доступность Query API | ≥ 99.5% (09:00–21:00 UTC) | SLA отчёт |
| NFR-3 | Доставка уведомлений | 99.9% за ≤10 сек от Firing | E2E замеры |
| NFR-4 | Ложные срабатывания | < 2% в неделю | Анализ инцидентов |
| NFR-5 | Ёмкость хранения | ≤ 2 TB на 1M рядов за 30 дней | Мониторинг диска |
| NFR-6 | Время восстановления (RTO) | ≤ 5 минут для Alert Evaluator | Инженерный тест |
| NFR-7 | Потеря данных (RPO) | ≤ 5 минут для конфигурации | Git-синхронизация |
| NFR-8 | Безопасность | TLS 1.3 для всех API | Сканер конфигурации |
| NFR-9 | Интерактивность UI | p95 < 5 сек для дашборда | Метрики Grafana |
| NFR-10 | Утилизация CPU | ≤ 70% на инстанс | Мониторинг |
| NFR-11 | Надёжность очереди | At-least-once доставка | Инжекция отказов |
